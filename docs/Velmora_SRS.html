Introduzione

1.1 Scopo del sistema

Velmora egrave; una piattaforma cross-platform (mobile e web) per la creazione di un diario di viaggio geolocalizzato multimediale. Il sistema consente agli utenti di tracciare percorsi GPS in tempo reale, registrare punti di interesse (memory marker) con contenuti multimediali associati (foto, video, audio, testo), visualizzare statistiche di viaggio e riprodurre il percorso sotto forma di replay animato su mappa interattiva.

Il nome Velmora egrave; una fusione concettuale che evoca viaggio + memoria, rappresentando fedelmente la mission del prodotto: trasformare ogni spostamento in un ricordo strutturato, navigabile e condivisibile.

1.2 Obiettivi

  1. Digitalizzare l’esperienza di viaggio come memoria strutturata e persistente;
  2. Integrare geolocalizzazione, multimedia e timeline in un’unica interfaccia coerente;
  3. Fornire una visualizzazione interattiva dei percorsi su mappa con Leaflet e OpenStreetMap;
  4. Supportare l’uso offline con sincronizzazione differita per garantire affidabilitagrave; in assenza di connettivitagrave;;
  5. Garantire scalabilitagrave; dell’architettura backend per supportare un numero crescente di utenti e tracce.

1.5 Ambito del progetto

Il sistema copre l’intero ciclo di vita di un viaggio: dalla creazione, al tracciamento GPS in tempo reale, alla registrazione di ricordi multimediali geolocalizzati, fino alla visualizzazione statistica e al replay. Non rientrano nell’ambito attuale: funzionalitagrave; social (following, feed pubblico), prenotazione di servizi turistici, integrazione con compagnie di viaggio o sistemi di pagamento.

2.3 Vincoli di progetto

  • V1 — Stack tecnologico: Frontend obbligatorio con Ionic Angular; backend in Node.js; database PostgreSQL con Prisma ORM; mappa con Leaflet + OpenStreetMap.
  • V2 — Cross-platform: L’applicazione deve essere eseguibile su iOS, Android e Web con la stessa base di codice.
  • V3 — Lingua: Interfaccia utente in italiano (con supporto futuro per inglese).
  • V4 — Open source: Il codice sorgente non egrave; necessariamente pubblico, ma l’architettura deve essere documentata per futuri contributori.
  • V5 — Containerizzazione: Il backend e il database devono essere containerizzati con Docker per riproducibilitagrave; dell’ambiente.

2.4 Assunzioni e dipendenze

  • A1: L’utente dispone di un dispositivo con sensore GPS e connettivitagrave; dati (almeno intermittente).
  • A2: L’utente acconsente all’uso della geolocalizzazione in background.
  • A3: I servizi esterni (OpenStreetMap, S3) sono disponibili e accessibili.
  • A4: Il backend egrave; deployato su una piattaforma cloud con supporto Docker (AWS ECS, Railway, o equivalente).
  • A5: Non egrave; richiesta conformitagrave; a regolamentazioni specifiche (es. GDPR oltre agli obblighi standard).

Architettura software

5.1 Architettura Three-Tier

Velmora adotta un’architettura Three-Tier (tre livelli), che separa la logica applicativa in tre strati distinti e indipendenti:

  1. Presentation Tier (Client Layer): Applicazione mobile/web realizzata con Ionic Angular. Gestisce l’interfaccia utente, la mappa Leaflet, l’interazione con l’utente e la logica di presentazione.
  2. Logic Tier (Application Layer): Backend API realizzato con Node.js (Express.js o NestJS). Contiene la logica di business, l’autenticazione, la validazione e l’orchestrazione dei dati.
  3. Data Tier (Data Layer): Database PostgreSQL con Prisma ORM per i dati strutturati, e object storage (S3) per i file multimediali.

6.4 Schema Prisma

Il seguente schema Prisma definisce il modello dati in modo dichiarativo. Prisma genera automaticamente il client TypeScript type-safe e le migrazioni SQL.

// schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique @db.VarChar(255)
  password  String   @db.VarChar(255)
  name      String?  @db.VarChar(100)
  avatarUrl String?  @map("avatar_url")
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")
  trips     Trip[]
  @@map("users")
}

model Trip {
  id          String    @id @default(uuid())
  userId      String    @map("user_id")
  title       String    @db.VarChar(200)
  description String?   @db.Text
  startDate   DateTime  @map("start_date")
  endDate     DateTime? @map("end_date")
  isActive    Boolean   @default(true) @map("is_active")
  createdAt   DateTime  @default(now()) @map("created_at")
  user        User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  points      GpsPoint[]
  markers     Marker[]
  @@map("trips")
}

model GpsPoint {
  id        String   @id @default(uuid())
  tripId    String   @map("trip_id")
  latitude  Float
  longitude Float
  timestamp DateTime
  speed     Float?
  accuracy  Float?
  altitude  Float?
  trip      Trip     @relation(fields: [tripId], references: [id], onDelete: Cascade)
  @@map("gps_points")
}

model Marker {
  id        String   @id @default(uuid())
  tripId    String   @map("trip_id")
  latitude  Float
  longitude Float
  type      String   @db.VarChar(20)
  content   String?  @db.Text
  mood      String?  @db.VarChar(50)
  createdAt DateTime @default(now()) @map("created_at")
  trip      Trip     @relation(fields: [tripId], references: [id], onDelete: Cascade)
  media     Media[]
  @@map("markers")
}

model Media {
  id        String   @id @default(uuid())
  markerId  String   @map("marker_id")
  fileUrl   String   @map("file_url")
  fileType  String   @db.VarChar(50) @map("file_type")
  fileSize  Int?     @map("file_size")
  metadata  Json?    @default("{}") @db.JsonB
  createdAt DateTime @default(now()) @map("created_at")
  marker    Marker   @relation(fields: [markerId], references: [id], onDelete: Cascade)
  @@map("media")
}

Manuale utente

1. Registrazione e login

  1. Apri l’app Velmora
  2. Tocca "Registrati" se non hai un account, inserisci email e password (minimo 8 caratteri, una maiuscola, un numero)
  3. Se hai giagrave; un account, inserisci le tue credenziali e tocca "Accedi"

2. Creare un viaggio

  1. Dalla schermata principale, tocca il pulsante "+" in basso a destra
  2. Inserisci un titolo per il viaggio (es. "Weekend a Roma")
  3. Tocca "Inizia viaggio"
  4. Il tracciamento GPS parte automaticamente. La mappa mostra la tua posizione in tempo reale.

3. Aggiungere un ricordo (marker)

  1. Durante il viaggio, tocca il pulsante "+" sulla mappa
  2. Scegli il tipo di contenuto: Foto, Video, Audio, Testo
  3. Scatta una foto, registra un audio, o scrivi un appunto
  4. Aggiungi opzionalmente un mood (felice, emozionato, ecc.)
  5. Conferma: il marker appare sulla mappa

4. Rivedere un viaggio

  1. Dalla lista viaggi, tocca un viaggio concluso
  2. Scegli tra: Mappa, Timeline, Statistiche, Replay
  3. Mappa: vedi il percorso con tutti i marker
  4. Timeline: sfoglia gli eventi in ordine cronologico
  5. Statistiche: distanza, tempo, velocitagrave;, marker
  6. Replay: guarda l’animazione del percorso

5. Modalitagrave; offline

  1. L’app funziona normalmente senza connessione
  2. I dati vengono salvati localmente
  3. Quando torni online, la sincronizzazione avviene automaticamente
  4. Un indicatore mostra lo stato della sincronizzazione

Conclusioni

Il presente documento ha descritto in dettaglio la specifica dei requisiti software per Velmora, una piattaforma cross-platform per il diario di viaggio geolocalizzato multimediale.

L’analisi ha coperto l’intero ciclo di vita del sistema:

  • Analisi dei requisiti: 40 requisiti funzionali e 20 non funzionali, coprendo autenticazione, gestione viaggi, tracciamento GPS, marker multimediali, mappe, timeline, statistiche, replay e offline mode.
  • Casi d’uso: 8 casi d’uso principali con flussi dettagliati.
  • Architettura: Three-Tier con pattern MVC + Repository, componenti, deployment e package.
  • Database: Modello relazionale normalizzato con 5 entitagrave;, schema Prisma completo, indici e vincoli.
  • API REST: 20+ endpoint documentati con formato richiesta/risposta.
  • Frontend: Architettura modulare Angular con integrazione Leaflet, wireframe e offline-first.
  • Sicurezza: JWT, bcrypt, HTTPS, rate limiting, protezione XSS/CSRF/SQL injection.
  • Test: Unit, integration, system e acceptance test pianificati.
  • DevOps: Docker, Docker Compose, GitHub Actions CI/CD, ambienti di deploy.

Lo stack tecnologico scelto (Ionic Angular + Node.js + PostgreSQL + Prisma + Leaflet) garantisce un equilibrio tra produttivitagrave; di sviluppo, performance e scalabilitagrave;. L’architettura offline-first assicura affidabilitagrave; in ogni condizione di connettivitagrave;.

Velmora si presenta come un sistema solido, ben documentato e pronto per essere implementato seguendo le specifiche qui descritte. La struttura modulare facilita futuri ampliamenti come funzionalitagrave; social, AI diary, AR overlay e condivisione viaggi.

Appendice A — Schema Prisma completo

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id             String   @id @default(uuid())
  email          String   @unique @db.VarChar(255)
  password       String   @db.VarChar(255)
  name           String?  @db.VarChar(100)
  avatarUrl      String?  @map("avatar_url")
  createdAt      DateTime @default(now()) @map("created_at")
  updatedAt      DateTime @updatedAt @map("updated_at")
  trips          Trip[]

  @@map("users")
}

model Trip {
  id             String    @id @default(uuid())
  userId         String    @map("user_id")
  title          String    @db.VarChar(200)
  description    String?   @db.Text
  startDate      DateTime  @map("start_date")
  endDate        DateTime? @map("end_date")
  isActive       Boolean   @default(true) @map("is_active")
  createdAt      DateTime  @default(now()) @map("created_at")
  user           User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  points         GpsPoint[]
  markers        Marker[]

  @@map("trips")
}

model GpsPoint {
  id             String   @id @default(uuid())
  tripId         String   @map("trip_id")
  latitude       Float
  longitude      Float
  timestamp      DateTime
  speed          Float?
  accuracy       Float?
  altitude       Float?
  trip           Trip     @relation(fields: [tripId], references: [id], onDelete: Cascade)

  @@index([tripId])
  @@index([tripId, timestamp])
  @@map("gps_points")
}

model Marker {
  id             String   @id @default(uuid())
  tripId         String   @map("trip_id")
  latitude       Float
  longitude      Float
  type           String   @db.VarChar(20)
  content        String?  @db.Text
  mood           String?  @db.VarChar(50)
  createdAt      DateTime @default(now()) @map("created_at")
  trip           Trip     @relation(fields: [tripId], references: [id], onDelete: Cascade)
  media          Media[]

  @@index([tripId])
  @@index([type])
  @@map("markers")
}

model Media {
  id             String   @id @default(uuid())
  markerId       String   @map("marker_id")
  fileUrl        String   @map("file_url")
  fileType       String   @db.VarChar(50) @map("file_type")
  fileSize       Int?     @map("file_size")
  metadata       Json?    @default("{}") @db.JsonB
  createdAt      DateTime @default(now()) @map("created_at")
  marker         Marker   @relation(fields: [markerId], references: [id], onDelete: Cascade)

  @@index([markerId])
  @@map("media")
}
